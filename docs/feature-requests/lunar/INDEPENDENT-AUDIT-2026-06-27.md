# Independent audit - LUNAR FR corpus - 2026-06-27

This is an independent, adversarial audit, a second pass over the 20 FRs of the LUNAR module, run after authoring went through 5 separate reviewers under the CyberOS `feature-request-audit` rubric. It supplements (does not replace) each FR's `*.audit.md` self-audit. Unlike self-audit, this pass was performed by reviewers who did NOT write the spec.

Method: 5 reviewers in parallel, each grading one cohesive cluster (001-003, 004-006, 007-010, 011-015, 016-020) together with its upstream FRs to catch cross-FR errors, checking the 11-section structure, §1 MUST -> §4 AC -> §5 test traceability, the correctness of the code/contract sketch, and fidelity to the PRD. A cross-cluster symbol sweep run by the parent (the convert/canChi function signatures, the Reminder type, FestivalContent) caught errors that straddle cluster boundaries where no single reviewer would see them.

## Headline

The independent pass found defects that self-audit missed: 2 blockers and about 13 code-level majors, plus a few minors. All blockers and code-level majors were fixed in the spec; minors were fixed when cheap, and the rest recorded as implementation notes. The author's self-audit was optimistic in declaring 10/10 across the board; this record corrects that and documents how each was resolved.

Two root causes dominate the bulk of the defects:

1. **Tuple-vs-object contract drift with `amlich-core`.** FR-LUNAR-001 returns a labeled tuple `[dd, mm, yy, leap]` for convert and a sentinel `[0, 0, 0]` for the invalid case. But FR-007 (grid render), FR-008 (`buildFestivalDateSet`), FR-010 (test mock), FR-016 (`todayLunar`), and FR-017 (`zns-scheduler`) all consume it as an object `.year` / `.month` / `.day`, producing `undefined` and `Invalid Date` at runtime; the sentinel `[0,0,0]` (a truthy array) is treated as `null`, so junk occurrences slip through. This is the kind of cross-FR error that a per-FR structural self-audit cannot see.

2. **Can-chi dia chi off by +8.** FR-LUNAR-002 (owner of can-chi) computes the day's can-chi with the canonical formula `can = (jdn + 9) % 10`, `chi = (jdn + 1) % 12`. FR-011 and FR-013 instead derive the dia chi as `((jdn + 9) % 60) % 12 = (jdn + 9) % 12`, off by +8 from FR-002. Because `THAN_TRUC_NHAT_TABLE`, Truc, and the Hoang dao hours are all keyed on the dia chi, the entire day-quality is wrong and contradicts the can-chi the calendar displays. (`can = (jdn+9)%10` happens to match, hiding the error.)

## Net

- Blockers: 2 (LUNAR-011, LUNAR-013 - can-chi dia chi +8) - both fixed, with a cross-consistency test vs `canChiDay`.
- Code-level majors: LUNAR-001 (x2), 004 (x2), 005, 006, 007 (x3), 008 (x2), 010 (x3), 013 (x2 beyond the blocker), 016 (x2), 017 (x2) - all fixed in the spec.
- Minor/NIT: fixed when cheap (002, 009, 012, 014, 015, 019 effort); the rest recorded as implementation notes.
- Most valuable catch: tuple-vs-object drift with amlich-core (wrong simultaneously in 007/008/010/016/017) and can-chi dia chi +8 (011/013). Both are cross-FR errors that no per-FR structural self-audit could see; one AC of FR-008 even contradicts its own test, and one fixture of FR-013 confuses the year can-chi with the day can-chi.

The corpus is clearly stronger after this pass. Mechanical re-verify after the fixes: 0 reciprocity errors, 0 stray typographic characters, full diacritics, every sub_task sum matches effort, total effort still 203h. A third pass is not needed before kickoff; re-audit an individual FR if its scope changes.

## Open items (do NOT auto-fix - need a founder/merge decision)

These are scope decisions or merge collisions, not a single FR's contract defect, so the reviewer records them instead of fixing them:

1. **ZNS MONTHLY recurrence (FR-017).** `SchedulerReminder` does not carry `recurrence`; the cron computes 1 occurrence/year, so MONTHLY reminders (Ram/Mung Mot) will NOT recur via ZNS server-side. Decision: support MONTHLY server-side (add `recurrence` + a month-expander loop) or scope ZNS to ANNUAL/ONCE only. This is worth locking before the P3 build.
2. **P3 migration-number collision.** `0016_*.sql` is used by both FR-017 and FR-018 in the same `services/genie-api/supabase/migrations/` namespace. Renumber on merge.
3. **Reminder subset in the client/scheduler (FR-016/017)** drops a few of FR-004's fields (`recurrence`/`sharedWith`); acceptable if computed on-the-fly, but it should carry `recurrence` for correct recurrence.
4. **FR-010 mirrors Reminder** and redeclares the type missing `notificationStyle`; the recommendation is to import `Reminder` from amlich-core instead of mirroring, to avoid drift.
5. **§5 traceability gaps** (a few ACs with no paired test) in FR-005/006 and scattered; add during implementation, not a contract defect.
