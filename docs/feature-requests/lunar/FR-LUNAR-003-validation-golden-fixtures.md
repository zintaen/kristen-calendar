## §1 - Description (BCP-14 normative)

The harness MUST prove the FR-LUNAR-001/002 engine matches the Ho Ngoc Duc calendar 100% for 1900-2199, including the edge years, and set a go/no-go threshold before any UI. Contract:

1. MUST encode the 6.6 fixtures table as fixed JSON, not generated by the engine itself; each row has the solar date, expected lunar date, can-chi year, and VN zodiac (NFR-Accuracy, DEC-LUNAR-030).
2. MUST assert the 5 Tet reference points from 6.6: 29/01/2025 -> 1/1/2025 Ất Tỵ; 17/02/2026 -> 1/1/2026 Bính Ngọ; 10/02/2024 -> 1/1/2024 Giáp Thìn; 22/01/2023 -> 1/1/2023 Quý Mão; 02/02/1984 -> 1/1/1984 Giáp Tý (PRD 6.6).
3. MUST assert the can-chi year and VN zodiac for each Tet reference point, where 2023 MUST yield the Cat (not the Rabbit) (FR-A03, PRD 6.6).
4. MUST assert 1985 has a leap month 2, and VN Tet 1985 = 21/01/1985 (one month earlier than China) (PRD 6.4, 6.6).
5. MUST assert three 21st-century years where VN differs from China - 2007, 2030, 2053 - by calling `convertSolar2Lunar` with `tz = 7.0` (VN) and `tz = 8.0` (China) and confirming the two countries' Tet differ by exactly 1 day (PRD Key Findings 2, DEC-LUNAR-032).
6. MUST run a full round-trip identity sweep 1900-2199: for every solar date in the range, `convertLunar2Solar(...convertSolar2Lunar(d, m, y, 7), 7)` MUST return exactly `(d, m, y)` (PRD 6.6, DEC-LUNAR-033).
7. MUST use a step small enough to cover every lunar month transition and leap month boundary; MUST NOT step so sparsely that it misses the first/last days of the month (DEC-LUNAR-033).
8. MUST apply the go/no-go threshold: any drift in any year in 1900-2199 fails the test and stops the build; MUST NOT loosen it to let a few years through (PRD Recommendations, DEC-LUNAR-031).
9. MUST read the fixtures from separate JSON files (`tet.json`, `vn-vs-china.json`), not embedding the expected values into the assert function itself, so the fixture is an independent source of truth (DEC-LUNAR-030).
10. MUST mark (log "suspect") the years with doubt around the Caveats points (for example the 28/9/2057 HKO warning) instead of silently swallowing the error (DEC-LUNAR-034, Caveats).
11. MUST run fully offline; the harness MUST NOT call network or compare against a runtime API (NFR-Offline).
12. MUST be the go/no-go gate for Phase 0: only when the whole harness is green may FR-LUNAR-007 and the later UI begin (DEC-LUNAR-031, PRD Phase 0).
13. SHOULD break out the assert counts clearly in the test report (number of Tet points, number of VN-vs-China years, sweep size) so QA can read the evidence.
14. SHOULD keep the fixtures extensible: adding a new row to the JSON does not require editing the assert code (DEC-LUNAR-030).
15. MUST compare the engine against an external golden dataset (`test/fixtures/gold-1900-2199.json`, exported from the Ho Ngoc Duc calculator or an equivalent source, not generated by the engine) including the edge years 1985/2007/2030/2053 and at least one representative historical sample, and assert exact solar<->lunar match; a bare round-trip is self-proving and not enough to confirm absolute accuracy (NFR-Accuracy, DEC-LUNAR-035). Commercial threshold (founder decision 3, DEC-LUNAR-039): exact match against the golden data is a hard gate for 1900-2100 (the range real users touch: past death anniversaries plus the near-future calendar); for 2100-2199 accept a discrepancy within 1 day, flag the suspect days per §1 #17 and fix them with a correction table if the comparison confirms a drift, WITHOUT blocking the gate.
16. MUST include an astronomy test set (dev-dependency only, NOT affecting the core's runtime dependency): load an ephemeris library (for example `astronomy-engine`) into `devDependencies`, compute the true Soc moment and true Solar Term at UTC+7, and confirm the engine's day result matches within 1 day at at least 120 sampled Soc points across 1900-2199; the goal is to test the Meeus approximation itself, catching the "Soc point near midnight" risk the PRD Caveats warn about (DEC-LUNAR-036).
17. MUST sweep all of 1900-2199 and detect every lunar-month start whose true Soc moment (from the ephemeris or the engine) is less than 15 minutes from midnight in Asia/Ho_Chi_Minh; emit that list as "suspect dates" and print it in the test report for manual review - it must not pass silently (DEC-LUNAR-037).
18. MUST run a set of property-based tests over the 1900-2199 range: (a) every lunar year has exactly 12 or 13 months; (b) the leap month appears only in a 13-month year; (c) lunar month 11 of each year contains the Dong chi (tietKhiAt returns isTrungKhi=true with name="Dong chi" within the days of lunar month 11); (d) each lunar month is 29 or 30 days long; (e) Tet (the 1st day of lunar month 1) is the 2nd Soc after the Dong chi (PRD 6.1 rule 3, Caveats, DEC-LUNAR-038).

## §2 - Why this design (rationale for humans)

**Why are the fixtures not generated by the engine itself?** If you take `convertSolar2Lunar`, produce a result, and then assert it against itself, the test is always green even when the engine is wrong - a meaningless self-proving loop. The fixtures must be independent values taken from the golden source (the Ho Ngoc Duc calendar, table 6.6), hard-coded into JSON, so the test genuinely reconciles the engine against the truth outside the engine (DEC-LUNAR-030).

**Why assert VN-vs-China with tz=7.0 and tz=8.0?** The VN/China difference is caused by the timezone (105E vs 120E), and PRD Key Findings 2 states plainly that the 21st century has exactly 3 years where Tet differs by 1 day: 2007, 2030, 2053. The most direct check is to call the same engine with two timezones and confirm it produces exactly that difference. If the engine gives the same result for both timezones in these years, it is not honoring rule 5 (DEC-LUNAR-032).

**Why a full round-trip sweep, not a sample?** A sample can miss exactly the year or exactly the month-boundary day where the engine is wrong. A full 1900-2199 sweep with a small step covers every month transition and every leap month; it is the cheapest and strongest safety net. An off-by-one in deriving k or an epoch drift will surface a wrong date on at least one day in the range (DEC-LUNAR-033).

**Why a hard go/no-go threshold?** The PRD Recommendations write it clearly: if any year drifts, stop and debug before building the UI. The lunar calendar is core trusted content; a reminder on the wrong date breaks the user's trust (Chi Linh). There is no "close enough" - either it matches 100% or it stops (DEC-LUNAR-031).

**Why mark suspect years instead of swallowing them?** The Caveats warn that the published algorithm is a simplified version; for a Soc day/tiet khi falling near midnight in a very distant year it may be off by 1 day (HKO warns around 28/9/2057). Instead of silently skipping it, the harness logs that year as "suspect" so it can be cross-checked against the golden source and a decision made, keeping the test set honest (DEC-LUNAR-034).

**Why put the fixtures in a separate JSON file?** Separating the data from the assert logic means adding a new reference point (for example Tet 2027, 2028) only requires editing the JSON, not touching the code. It also makes clear which part is the independent truth and which is the check function, true to the spirit of DEC-LUNAR-030.

**Why an external golden dataset (gold-1900-2199.json)?** The round-trip `L2S(S2L(d))` only proves the engine's internal consistency - if the engine has a systematic error (for example an epoch off by a large constant), the round-trip is still green. To confirm correctness against the actual VN calendar rules, there must be at least one sample of data from a source external to the engine (the Ho Ngoc Duc calculator). This is the difference between testing consistency and testing absolute accuracy (DEC-LUNAR-035).

**Why is the astronomy oracle a dev-dependency rather than in the core?** The core MUST be zero-dependency and offline; that is a hard requirement of FR-LUNAR-001. The ephemeris library is a dev-tool used only in the CI environment, never bundled into the app. The goal is to test how good the Meeus approximation itself is, especially at Soc points falling near midnight - where the approximation error can push the day off by 1 unit (DEC-LUNAR-036, PRD Caveats).

**Why a suspect-midnight report instead of auto-fixing?** If the engine fixes itself silently, the operator does not know where the risk is. Emitting the list lets the founder and QA spot-check those days manually and decide whether a correction table is needed. HKO has warned specifically about 28/9/2057; having the full list helps prepare before that day arrives (DEC-LUNAR-037).

**Why do property-based tests complement the fixtures?** The fixtures check specific days; the property tests check the structure of the whole 1900-2199 calendar. An engine can pass the 5 Tet points but return a lunar year with 14 months in some year. The property tests catch that structural error before it produces an impossible calendar for the user (DEC-LUNAR-038).

## §4 - Acceptance criteria

1. Each row in `tet.json`: `convertSolar2Lunar(...solar, 7)` equals the row's `lunar` (5 Tet points, PRD 6.6).
2. Each row in `tet.json`: `canChiYear(lunar[2]).label` equals the row's `canChiYear`.
3. The 2023 row in `tet.json`: `zodiacOf(2023)` equals `Mèo` (not the Rabbit).
4. Reverse each Tet point: `convertLunar2Solar(...lunar, 7)` equals the row's `solar`.
5. 1985: `convertLunar2Solar(1, 1, 1985, 0, 7)` equals `[21, 1, 1985]` (VN Tet, PRD 6.4).
6. 1985 leap month 2: `convertLunar2Solar(1, 2, 1985, 1, 7)` differs from `[0, 0, 0]` and falls in the window `[[21,3,1985],[19,4,1985]]`.
7. 2007: VN Tet (`tz=7`) and China Tet (`tz=8`) differ by exactly 1 day (PRD Key Findings 2).
8. 2030: VN Tet and China Tet differ by exactly 1 day.
9. 2053: VN Tet and China Tet differ by exactly 1 day.
10. Full round-trip identity sweep 1900-2199: for every solar date in the range, `convertLunar2Solar(...convertSolar2Lunar(d, m, y, 7), 7)` equals `(d, m, y)`; 0 drift.
11. The sweep counts the number of days checked and prints it (evidence), with the count > 109,000 days (roughly 300 years).
12. Go/no-go threshold: if any assert fails, the suite fails with a non-zero exit code (no loosening).
13. The fixtures are read from JSON, not embedding the expected values into the assert function (DEC-LUNAR-030).
14. The harness does not call network during the whole test (offline, NFR-Offline).
15. Suspect years (if any) are logged as "suspect" rather than swallowed silently (DEC-LUNAR-034).
16. `absolute-ground-truth.test.ts` loads `gold-1900-2199.json`, walks each row, and asserts `convertSolar2Lunar(...solar, 7)` matches the row's `lunar` exactly; 0 drifting rows is a pass (DEC-LUNAR-035).
17. `gold-1900-2199.json` includes at least the edge years 1985 (leap month 2 and VN Tet 21/01), 2007, 2030, 2053 (Tet differing from China by 1 day) and at least 30 representative sampled days across the whole 1900-2199 range; this fixture MUST NOT be generated by the engine itself (DEC-LUNAR-035).
18. `astronomy-oracle.test.ts` runs in the `devDependencies` context (not affecting the production bundle); with at least 120 sampled Soc points across 1900-2199, assert `|engineDay - oracleDay|` = 0 for >= 95% of the samples and <= 1 day for 100% of the samples; print the number of drifting points and the list of drifting days (DEC-LUNAR-036).
19. `suspect-midnight.test.ts` emits the list of all lunar-month-start days in 1900-2199 whose Soc point is less than 15 minutes from midnight in Asia/Ho_Chi_Minh; the list is printed to test output but does not fail the test; 28/9/2057 (the HKO watch point) must appear in the list if the calculation shows it within the threshold (DEC-LUNAR-037).
20. `property.test.ts` sweeps 1900-2199 and asserts: (a) every lunar year has 12 or 13 months, with no year outside these two values; (b) in the 12-month years, there is no leap month; (c) each year has at least one day in lunar month 11 for which tietKhiAt returns the Dong chi; (d) each lunar month has 29 or 30 days; 0 violations is a pass (DEC-LUNAR-038).
21. `property.test.ts` also asserts that Tet (the 1st day of lunar month 1) with convertLunar2Solar(1, 1, y, 0, 7) matches the 2nd Soc after the Dong chi (that is, after getLunarMonth11(y-1, 7)) with 0 days tolerance, across at least 5 representative Tet years including 2024/2025/2026 (DEC-LUNAR-038, PRD 6.1 rule 3).

```typescript
// packages/amlich-core/test/golden.test.ts
import { describe, it, expect } from "vitest";
import tet from "./fixtures/tet.json";
import vnCn from "./fixtures/vn-vs-china.json";
import { convertSolar2Lunar, convertLunar2Solar, jdFromDate, VN_TIMEZONE } from "../src/index";
import { canChiYear } from "../src/canchi";
import { zodiacOf } from "../src/zodiac";

const VN = VN_TIMEZONE;
const CN = 8.0;

describe("Golden Tet fixtures (PRD 6.6, doc tu tet.json)", () => {
  for (const row of tet) {
    const [d, m, y] = row.solar;
    it(`${d}/${m}/${y} -> ${row.lunar.join("/")} (${row.note})`, () => {
      expect(convertSolar2Lunar(d, m, y, VN)).toEqual(row.lunar);
      expect(canChiYear(row.lunar[2]).label).toBe(row.canChiYear);
      expect(zodiacOf(row.lunar[2])).toBe(row.zodiac);
      expect(convertLunar2Solar(row.lunar[0], row.lunar[1], row.lunar[2], row.lunar[3] as 0 | 1, VN))
        .toEqual(row.solar);
    });
  }
});

describe("VN-vs-China offsets (tz=7 vs tz=8, PRD Key Findings 2 + 6.4)", () => {
  it("Tet VN 1985 = 21/01/1985 (som hon TQ mot thang)", () => {
    const [d, m, y] = vnCn.tetVN1985.solarVN;
    expect(convertLunar2Solar(1, 1, 1985, 0, VN)).toEqual([d, m, y]);
  });

  it("thang 2 nhuan 1985 roi trong cua so 21/03..19/04", () => {
    const s = convertLunar2Solar(1, 2, 1985, 1, VN);
    expect(s).not.toEqual([0, 0, 0]);
    const jd = jdFromDate(s[0], s[1], s[2]);
    const [lo, hi] = vnCn.leap1985.windowVN;
    expect(jd).toBeGreaterThanOrEqual(jdFromDate(lo[0], lo[1], lo[2]));
    expect(jd).toBeLessThanOrEqual(jdFromDate(hi[0], hi[1], hi[2]));
  });

  for (const row of vnCn.offsetDays) {
    it(`Tet ${row.year}: VN vs TQ lech ${row.expectDiffDays} ngay (${row.note})`, () => {
      const vn = convertLunar2Solar(1, 1, row.year, 0, VN);
      const cn = convertLunar2Solar(1, 1, row.year, 0, CN);
      const diff = Math.abs(jdFromDate(vn[0], vn[1], vn[2]) - jdFromDate(cn[0], cn[1], cn[2]));
      expect(diff).toBe(row.expectDiffDays);
    });
  }
});
```

```typescript
// packages/amlich-core/test/roundtrip.test.ts
import { describe, it, expect } from "vitest";
import { convertSolar2Lunar, convertLunar2Solar, jdFromDate, jdToDate, VN_TIMEZONE } from "../src/index";

const VN = VN_TIMEZONE;

describe("Round-trip identity sweep 1900-2199 (go/no-go gate, DEC-LUNAR-033)", () => {
  it("L2S(S2L(d,m,y)) == (d,m,y) cho MOI ngay trong dai", () => {
    const start = jdFromDate(1, 1, 1900);
    const end = jdFromDate(31, 12, 2199);
    let checked = 0;
    const suspects: string[] = [];
    for (let jd = start; jd <= end; jd += 1) {     // bước 1 ngày: phủ hết ranh tháng
      const [d, m, y] = jdToDate(jd);
      const lunar = convertSolar2Lunar(d, m, y, VN);
      const back = convertLunar2Solar(lunar[0], lunar[1], lunar[2], lunar[3], VN);
      if (back[0] !== d || back[1] !== m || back[2] !== y) {
        suspects.push(`${d}/${m}/${y} -> ${lunar.join("/")} -> ${back.join("/")}`);
      }
      checked++;
    }
    if (suspects.length) console.warn("SUSPECT round-trip mismatches:", suspects.slice(0, 20));
    expect(suspects).toEqual([]);                  // 0 lệch: ngưỡng go/no-go
    expect(checked).toBeGreaterThan(109_000);      // evidence: ~300 năm
  });
});
```

## §6 - Implementation skeleton

The harness in §5 is a full skeleton. The detail to pin: the round-trip sweep uses a 1-day step (`jd += 1`) rather than a sample, because an off-by-one only surfaces on exactly the first or last day of the lunar month - a sparse sample would miss it. A full 1900-2199 sweep runs in a few seconds thanks to the NFR-Performance of < 5ms per conversion, and this is an acceptable price for a go/no-go gate running in CI. The VN-vs-China asserts call the same engine with `tz=7.0` and `tz=8.0`; the difference in results is itself the evidence that the engine honors rule 5.

## §11 - Implementation notes

- The fixture is the independent source of truth; take it from table 6.6 and the Ho Ngoc Duc site, and encode it into `tet.json` and `vn-vs-china.json`. Never generate the expected values from `convertSolar2Lunar` itself - that is a self-proving loop and will let every error through.
- The round-trip sweep uses a 1-day step (`jd += 1`) fully across 1900-2199. An off-by-one in deriving k or an epoch drift only surfaces on exactly the first or last day of the lunar month; a sparse sample would miss exactly those days.
- VN-vs-China is checked by calling the same engine with `tz=7.0` and `tz=8.0`. PRD Key Findings 2 fixes 3 21st-century years differing by 1 day (2007, 2030, 2053); if the engine gives the same Tet for both timezones in these years, it is not honoring rule 5.
- The go/no-go threshold is hard: 0 drift or stop the build. The PRD Recommendations write it clearly - any drift in any year means stop and debug before building the UI. There is no "close enough".
- The commercial threshold per founder decision 3 (DEC-LUNAR-039): the hard gate of an exact match against the golden data applies to 1900-2100, the range users actually touch (past ancestral death anniversaries plus near-future reminders and festivals). For 2100-2199, the Meeus approximation may be off by 1 day at a few Soc points near midnight; these days are flagged (§1 #17) and fixed with a small correction table if the comparison confirms a drift, without blocking the gate. The round-trip sweep stays hard 1900-2199 because it is free and catches off-by-ones everywhere.
- Log suspect years instead of swallowing them: the Caveats warn the simplified algorithm can be off by 1 day in a very distant year (HKO around 28/9/2057, China standard). The harness prints suspect for cross-checking and a decision, keeping the test set honest.
- Print the number of days checked and the number of fixture points as evidence for QA: a line "sweep 1900-2199, checked 109573, mismatches 0, verdict GO" is the evidence for the Phase 0 release.
- Separate the fixtures into their own JSON so adding a new reference point (Tet 2027, 2028, or a new reconciliation year) only requires editing the data, not touching the assert code - true to the spirit of DEC-LUNAR-030 and easy to maintain.
- `gold-1900-2199.json` is an asset external to the engine. Suggested ways to create it: (1) run a batch script on the Ho Ngoc Duc calculator (informatik.uni-leipzig.de/~duc/amlich or the mirror xemamlich.uhm.vn) to export the calendar day by day; (2) use `vanng822/amlich` (Node, MIT) in a single run with a different engine; (3) aggregate at least the required rows: the 5 Tet points from 6.6, all 4 edge years (1985/2007/2030/2053), and a sampling of >= 30 representative days. This file MUST NOT be created by the engine under test.
- The improvement path if the astronomy oracle finds a 1-day drift: add a small correction table (`src/correction.ts`) containing the specific days to override, keeping the core offline and zero-dependency; the override applies only when the engine returns a value different from the table; record the reference source (HKO, Duc) for each override row. This is the commercial path if `suspect-midnight.test.ts` finds a day under 15 minutes that the oracle confirms is drifting. The point to watch first: HKO warns about 28/9/2057 (China standard 120E; the VN 105E impact needs confirmation).
